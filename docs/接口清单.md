# 接口与端口清单（默认 `BASE_PORT=9000`）

这份文档面向两类读者：
- 人工联调人员：快速查端口、路径、方法、返回结果
- AI / 自动化工具：直接据此生成请求、断言返回字段、理解静态资源来源

项目启动后会同时开启 6 个端口：
- HTTP 服务（3 个）：`BASE_PORT`、`BASE_PORT+1`、`BASE_PORT+2`
  - 默认分别为 `9000`、`9001`、`9002`
- WebSocket 服务（3 个）：`BASE_PORT+3`、`BASE_PORT+4`、`BASE_PORT+5`
  - 默认分别为 `9003`、`9004`、`9005`

可通过环境变量 `BASE_PORT` 覆盖基础端口。所有 HTTP JSON 响应的 `Content-Type` 均为 `application/json; charset=utf-8`。

---

## 1. 统一规则

### 1.1 静态资源目录

- 默认目录：`assets/`
- 可通过 `ASSETS_DIR` 环境变量覆盖
- 包内测试场景会自动探测 `../../assets`

### 1.2 WebSocket 鉴权

- 静态令牌：`zhongmiao-org-token`
- 满足以下任一方式即可：
  - Header：`X-Auth-Token: zhongmiao-org-token`
  - Query：`?token=zhongmiao-org-token`
- 未提供或错误时返回：

```json
{
  "error": "unauthorized",
  "hint": "provide X-Auth-Token header or ?token="
}
```

### 1.3 WS 事件键名规则

- `ws-echo`：事件键名为 `type`
- `ws-ticker`：事件键名为 `action`
- `ws-timeline`：事件键名为 `event`

### 1.4 多 route / stripPrefix 推荐用途

当前 HTTP 服务除了原始前缀路径，还额外提供了一批“根路径别名”接口，方便验证以下代理场景：
- `stripPrefix=true`
- `/` 兜底 route + `/api` 子 route
- `/api` 与 `/api/admin` 的最长前缀优先匹配
- 不同 route 指向不同 upstream 服务

## 2. HTTP 通用接口（9000 / 9001 / 9002 全部支持）

以下接口在 3 个 HTTP 服务上完全一致，只是根路径返回的 `service`、`port`、`interceptPrefix` 不同。

### 2.1 根信息

- `GET /`
- 示例：
  - `http://localhost:9000/`
  - `http://localhost:9001/`
  - `http://localhost:9002/`
- 返回结构：

```json
{
  "service": "user-service",
  "port": 9000,
  "interceptPrefix": "/api",
  "message": "Upstream running"
}
```

不同端口对应值：
- 9000：`service=user-service`，`interceptPrefix=/api`
- 9001：`service=order-service`，`interceptPrefix=/order-api`
- 9002：`service=payment-service`，`interceptPrefix=/pay-api`

### 2.2 健康检查

- `GET /health`
- 返回：

```json
{
  "status": "ok"
}
```

### 2.3 指定状态码

- `GET /status/{code}`
- 例：`GET /status/418`
- 行为：
  - `code` 在 `100..599` 范围内时，HTTP 状态码就是该值
  - 非法值回退为 `400`
- 返回：

```json
{
  "status": 418
}
```

### 2.4 延迟响应

- `GET /delay/{ms}`
- 例：`GET /delay/500`
- 行为：服务端延迟指定毫秒后返回
- 返回：

```json
{
  "delayedMs": 500
}
```

### 2.5 请求头回显

- `GET /headers`
- 仅回显存在的这些请求头：
  - `Authorization`
  - `Content-Type`
  - `User-Agent`
  - `X-Request-Id`
- 返回示例：

```json
{
  "headers": {
    "Authorization": "Bearer demo",
    "X-Request-Id": "req-001"
  }
}
```

### 2.6 Cookie 回显

- `GET /cookies`
- 返回示例：

```json
{
  "cookies": {
    "sid": "abc",
    "user": "tom"
  }
}
```

### 2.7 大包响应

- `GET /large?size=<n>`
- 默认 `n=65536`
- 最大 `2MB`
- 返回结构：

```json
{
  "size": 64,
  "data": "aaaaaaaaaaaaaaaa..."
}
```

### 2.8 Echo

- `POST /echo`
- `PUT /echo`
- `PATCH /echo`
- 其他方法也会被处理，但通常建议使用以上 3 个方法
- 返回结构：

```json
{
  "method": "POST",
  "path": "/echo",
  "query": "a=1&b=2",
  "length": 13,
  "body": "{\"name\":\"abc\"}"
}
```

### 2.9 RESTful 集合接口

- `GET /rest/items`
  - 返回种子数据或运行期内存数据：

```json
{
  "items": [
    {
      "id": 1,
      "name": "Alpha",
      "type": "service",
      "enabled": true
    }
  ]
}
```

- `POST /rest/items`
  - 请求体：任意 JSON 对象
  - 返回：`201 Created`
  - Header：`Location: /rest/items/{id}`
  - 返回体示例：

```json
{
  "id": 4,
  "name": "New Item"
}
```

- `OPTIONS /rest/items`
  - 返回：`204 No Content`
  - Header：`Allow: GET,POST,OPTIONS`

### 2.10 RESTful 单项接口

- `GET /rest/items/{id}`
  - 成功返回对象
  - 不存在返回：

```json
{
  "error": "not found"
}
```

- `PUT /rest/items/{id}`
  - 全量替换对象
  - 返回更新后的对象

- `PATCH /rest/items/{id}`
  - 部分更新对象
  - 返回更新后的对象

- `DELETE /rest/items/{id}`
  - 返回：`204 No Content`

- `OPTIONS /rest/items/{id}`
  - 返回：`204 No Content`
  - Header：`Allow: GET,PUT,PATCH,DELETE,OPTIONS`

- `id` 非法时返回：

```json
{
  "error": "invalid id"
}
```

## 4. 9001：order-service

### 4.2 根路径别名接口

适合 `stripPrefix=true`：
- `GET /orders`
- `POST /orders`
- `GET /orders/{id}`
- `GET /admin/orders/summary`
- `GET /order/{id}/submit`

### 4.3 返回结果

#### `GET /order-api/orders` 或 `GET /orders`

数据来源：`assets/order/orders.json`

返回示例：

```json
{
  "code": 0,
  "data": [
    {
      "id": 2001,
      "status": "CREATED",
      "amount": 49.9,
      "currency": "CNY"
    },
    {
      "id": 2002,
      "status": "PAID",
      "amount": 199,
      "currency": "CNY"
    }
  ]
}
```

#### `POST /order-api/orders` 或 `POST /orders`

请求体：任意 JSON 对象

动态规则：
- 服务端会向 `data` 注入随机 `id`
- HTTP 状态码固定为 `201`

请求示例：

```json
{
  "sku": "A-1",
  "qty": 2
}
```

返回示例：

```json
{
  "code": 0,
  "data": {
    "sku": "A-1",
    "qty": 2,
    "id": 54321
  }
}
```

#### `GET /order-api/orders/{id}` 或 `GET /orders/{id}`

数据来源：`assets/order/detail.json`

动态规则：
- 返回体中的 `data.id` 会替换为请求路径中的 `{id}`
- 若 `items[0].sku` 存在，也会同步更新为 `SKU-{id}`

例：`GET /orders/3009`

返回示例：

```json
{
  "code": 0,
  "data": {
    "id": "3009",
    "status": "PROCESSING",
    "currency": "CNY",
    "amount": 88.5,
    "items": [
      {
        "sku": "SKU-3009",
        "qty": 1,
        "price": 59.9
      }
    ]
  }
}
```

#### `GET /order-api/admin/orders/summary` 或 `GET /admin/orders/summary`

数据来源：`assets/order/admin_summary.json`

返回示例：

```json
{
  "code": 0,
  "data": {
    "created": 12,
    "paid": 9,
    "cancelled": 3,
    "avgAmount": 132.45
  }
}
```

#### `GET /order-api/order/{id}/submit` 或 `GET /order/{id}/submit`

数据来源：`assets/order/submit.json`

说明：
- 只要路径以 `/submit` 结尾即命中
- 不满足该格式返回 `404`

返回示例：

```json
{
  "code": 0,
  "message": "submit ok",
  "data": {
    "accepted": true,
    "queue": "fulfillment",
    "etaMinutes": 15
  }
}
```

## 6. 9003：ws-echo

根信息：
- `GET http://localhost:9003/`
- 返回：

```json
{
  "service": "ws-echo",
  "port": 9003
}
```

### 6.1 `WS /ws/echo`

- 地址：`ws://localhost:9003/ws/echo?token=zhongmiao-org-token`

- 地址：`ws://localhost:9003/ws/food/user?token=zhongmiao-org-token[&interval=ms]`

```json
{
  "type": "order_accepted",
  "orderId": "123456",
  "riderId": "rider_888",
  "riderName": "张师傅"
}
```

- 默认间隔：`450ms`

- 数据来源：`assets/ws/food_merchant.json`
- 结果：按顺序推送商家侧事件，然后正常关闭连接

首条示例：

## 10. 推荐验证用例

### 10.1 `/` + `/api`

- route A：`/` → `http://localhost:9000`
- route B：`/api` → `http://localhost:9001`，`stripPrefix=true`

可验证：
- `/` 命中前端兜底
- `/api/orders` 转发到 `http://localhost:9001/orders`
- `/api/admin/orders/summary` 转发到 `http://localhost:9001/admin/orders/summary`

### 10.2 `/api` + `/api/admin`

- route A：`/api` → `http://localhost:9000`，`stripPrefix=true`
- route B：`/api/admin` → `http://localhost:9002`，`stripPrefix=true`

可验证：
- `/api/users` 命中 user-service
- `/api/admin/callbacks/alipay` 命中 payment-service
- 最长前缀 `/api/admin` 优先于 `/api`

### 10.3 Mock 与转发混合

- 对 `/api/orders/3009` 使用 upstream 转发
- 对 `/api/orders/3009` 或 `/api/admin/stats` 在代理层配置 mock
- 可对比真实 upstream 返回与代理层 mock 返回是否符合预期

## 11. 备注

- 9000 / 9001 / 9002 共享通用调试接口，所以 path 会在不同端口重复出现。
- 根路径别名接口是为了更方便测试多 route 与 `stripPrefix=true`，不影响原有前缀路径接口。
- RESTful 数据在服务启动时会从 `assets/rest/items.json` 初始化到内存；后续增删改只影响进程内存，不会回写文件。
- 如需修改基础端口，请在启动前设置：`BASE_PORT=<起始端口>`。
